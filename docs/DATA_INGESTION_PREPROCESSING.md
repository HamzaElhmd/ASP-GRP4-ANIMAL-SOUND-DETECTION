# Data Ingestion & Preprocessing

## Scope

This document explains the implementation completed for Project 1, Part 1: Data Ingestion & Preprocessing, in `src/audit.py` and `src/preprocessor.py`.

The goal of this section is to provide a shared, consistent audio pipeline for the rest of the project:

1. audit the raw corpus,
2. generate a metadata manifest,
3. compute class-level dataset statistics,
4. standardize every clip into a common audio format for downstream modeling.

## Raw Dataset Audit

The raw dataset is stored under `dataset/`, with one subdirectory per label. The audit logic scans each class folder, loads each audio file with `torchaudio.load()`, and extracts content-based metadata directly from the waveform tensor.

### Metadata captured per clip

The manifest generated by `create_dataset_metadata()` stores:

- `filepath`: original file path
- `n_samples`: number of waveform samples
- `channel`: `mono` or `stereo`
- `samplerate`: native sample rate
- `duration`: clip duration in seconds
- `label`: class name derived from the folder

This gives the team a single, reusable CSV describing the dataset without relying on filename conventions.

### Corpus findings

Running the audit on the current corpus shows:

- `908` total clips
- class imbalance across the six labels
- mixed sample rates
- both mono and stereo files
- highly variable durations

Class counts:

- `cat`: 240

- `background`: 150
- `cow`: 115
- `sheep`: 83

The duration range is broad. Some clips are under one second, while others go up to 60 seconds. That variability makes raw training impossible without normalization and preprocessing.

The dataset also contains several sample rates, including 8 kHz, 11.025 kHz, 16 kHz, 22.05 kHz, 24 kHz, 44.1 kHz, 48 kHz, and a few irregular rates from source files. This is exactly the kind of heterogeneity the preprocessing step is supposed to remove.

## Preprocessing Design

The preprocessing logic is implemented in `preprocess_audio()` and `preprocess_dataset()`.

The design follows the spec requirement for standardization:

- convert stereo to mono,
- resample every clip to a fixed target sample rate,
- peak-normalize the waveform,
- save the processed output and its updated metadata.

The pipeline was then extended with three additional pieces required by the spec:

- offline framing for training data,
- online sliding-window inference for long recordings,
- offline augmentation with background overlap.

## Step 1: Mono Conversion

If the clip is stereo, the two channels are averaged with `torch.mean(audio, dim=0, keepdim=True)`.

This preserves the overall energy profile while collapsing the waveform to a single channel. If the clip is already mono, it is left unchanged.

Why this matters:

- it makes the input shape uniform,
- it avoids channel-specific variance between sources,
- it simplifies downstream feature extraction and framing.

## Step 2: Fixed Sample Rate

All audio is standardized to `16 kHz`.

The initial version of the code only downsampled clips with a higher sample rate and left lower-rate files unchanged. That would have produced a mixed-rate processed corpus, which is not acceptable for a shared dataset pipeline.

The implementation now resamples any clip whose rate is not the target rate using `torchaudio.transforms.Resample`.

Why `16 kHz`:

- it is a common audio ML default,
- it keeps compute and storage manageable,
- it is sufficient for animal sound event detection without preserving unnecessary high-frequency detail.

## Step 3: Peak Normalization

After resampling, the waveform is peak-normalized with:

```text
y' = y / max(|y|)
```

This removes amplitude-scale bias across recordings from different sources and recording conditions.

Silent clips are handled safely. If the maximum absolute amplitude is zero, the code skips the division instead of dividing by zero.

## Step 4: Processed Output

Each output path is rewritten into:

```text
processed/<label>/<filename>
```

The implementation also creates missing directories before saving.

In addition to the waveform files, the processed metadata is written to:

```text
processed/farmyard.csv
```

This gives the rest of the project a ready-to-use manifest for training, evaluation, or feature extraction.
To keep memory usage bounded on large runs, the processed manifest is written incrementally to disk instead of being accumulated entirely in RAM.

## Training Framing

The spec asks for fixed-length training windows that are much shorter than a full recording but still long enough to contain a vocalization. This is implemented in `frame_training_audio()`.

### Chosen framing policy

- window length: `2.0 s`
- sample rate: `16 kHz`
- samples per window: `32,000`
- padding: zero-pad the last frame when the clip does not divide evenly

### Why 2 seconds

Two seconds is a practical compromise for this dataset:

- it is short enough to create many training samples from long clips,
- it is long enough to preserve the shape of most animal vocalizations,
- it gives the classical ML track consistent frame-level inputs,
- it avoids forcing the model to learn from full-length recordings that are too variable.

### Framing behavior

The function:

- loads and standardizes the clip,
- splits it into consecutive fixed windows,
- pads the final window when needed,
- stores each frame as a separate offline sample,
- records segment metadata in the output manifest.

The framed samples are saved under:

```text
processed/<label>/<source_name>_frame_<index>.wav
```

This is an offline preprocessing step because training needs a static labeled dataset, not an on-the-fly scan.

Augmented clips are framed with the same window size so the final training corpus remains shape-consistent. In the saved filenames, those examples follow the pattern:

```text
processed/<label>/<source_name>_aug_<k>_frame_<index>.wav
```

That makes it explicit that the sample is both augmented and framed.

## Inference-Time Windowing

Inference-time windowing is intentionally kept separate from preprocessing and is implemented in `inference_time_windowing()`.

### Chosen scan policy

- window length: `1.0 s`
- hop length: `0.25 s`
- overlap: `75%`
- window function: Hann

### Why this choice

The spec explicitly asks for high temporal resolution during inference. A 1-second window with a 250 ms hop gives dense coverage of the unseen recording while still being long enough to capture a local acoustic event.

The tradeoff is standard:

- shorter windows improve temporal localization,
- longer windows improve frequency resolution.

For this project, the inference problem is event detection on farm recordings, so we prioritize temporal resolution. That makes it easier to catch the start and end of short animal vocalizations and to locate overlapping events precisely.

The Hann window reduces spectral leakage at frame boundaries and is a reasonable default for scan-based audio analysis.

### Why it stays online

This function is not part of offline preprocessing because it is intended to be called during EDA and modeling on long recordings. Keeping it online avoids baking a single scan strategy into the processed dataset and leaves the downstream phases free to reuse the same raw standardized clips with different model settings.

## Augmentation

The augmentation logic is implemented in `augment_audio()`. It is designed to compensate for the small seed set while staying faithful to the project goal: detect animal sounds in a farm environment, not isolated clean clips.

### Chosen augmentation operators

For each non-background clip, the function can apply:

- background overlap,
- time shifting,
- gain variation,
- optional noise injection.

### Background overlap

This is the most important augmentation choice for the project.

The goal is not just to classify clean animal sounds. The model must detect labeled animals in the presence of farm ambience, so augmented animal clips should often contain background audio.

The implementation therefore samples a `background` clip and mixes it into the target clip with a random gain. This creates harder examples that more closely resemble the intended deployment setting.

### Additional transforms

After background mixing, the function applies:

- a random circular time shift,
- a random gain change in dB,
- optional additive noise.

These operations simulate small temporal offsets, microphone variation, and recording noise. They are deliberately lightweight so they preserve the label while increasing diversity.

### Label policy

The label of the augmented sample remains the original animal class. That is the correct choice here because the background is not the target class; it is contextual clutter that should not overwrite the animal label.

### Before/after class balance

The raw corpus class counts are:

- `cat`: 240
- `dog`: 240
- `background`: 150
- `cow`: 115
- `sheep`: 83
- `rooster`: 80

The offline preprocessing pipeline now creates:

- framed windows for every source clip,
- additional augmented copies for each source clip,
- background-mixed variants for non-background labels,
- framed outputs for every augmented sample as well.

That means the processed corpus is substantially larger than the raw dataset and less imbalanced than the original seed set because every rare class receives extra synthetic coverage. The exact final balance depends on the augmentation multiplier, which is currently controlled by `DEFAULT_AUGMENTATIONS_PER_CLIP`.

For the current implementation:

- each original clip contributes framed windows,
- each original clip also contributes two augmented versions by default,
- each augmented version is itself split into the same fixed-length training windows,
- the total label counts increase uniformly at the source level,
- background audio is injected into animal examples to make the minority classes more robust in realistic conditions.

## Implementation Details

### `src/audit.py`

The audit module now provides:

- `dataset_info()`: per-class counts and total duration
- `create_dataset_metadata()`: raw audio scan and manifest creation
- `dataset_statistics()`: class-level aggregation over the manifest

The metadata CSV is now saved with `index=False`, which keeps the file clean and easier to reuse.

### `src/preprocessor.py`

The preprocessing module now:

- loads `dataset/farmyard.csv`,
- reads each waveform,
- converts stereo to mono,
- resamples every clip to `16 kHz`,
- peak-normalizes the waveform,
- writes the processed clip to `processed/<label>/<filename>`,
- saves the updated manifest to `processed/farmyard.csv`.

The output directories are created automatically, so the script runs cleanly from a fresh checkout.

## Issues Fixed From the Initial Version

The original implementation had several problems that would have broken the pipeline or produced inconsistent output:

- the sample-rate column was accessed incorrectly with `dataset_df.loc["samplerate"]`
- the preprocessing rate logic only handled downsampling, not full standardization
- the output filename was extracted with a brittle string split on `/`
- the processed directory tree was not created before saving
- the processed manifest was constructed but not persisted
- peak normalization could divide by zero on silent audio
- the raw metadata CSV was written with the default index column

These issues are now corrected.

## Methodology Rationale

The preprocessing choices are intentionally conservative.

The point of this section is not to invent sophisticated transformations. It is to build a stable input base shared by all later model tracks.

Mono conversion and resampling remove format variance. Peak normalization removes amplitude bias. The metadata and statistics step provides the evidence needed to justify the preprocessing decisions in the report and presentation.

Framing and augmentation are the two additional steps that make the corpus usable for training:

- framing creates fixed-size supervised examples,
- augmentation expands the dataset and makes it more realistic,
- augmented samples are framed too, so the final training corpus stays uniform,
- background overlap ensures the model learns animal sounds in context rather than only in isolation.

This foundation is compatible with the rest of the spec:

- classical ML can use the manifest to derive fixed windows and features,
- CNN-based models can use the standardized waveform or a spectrogram derived from it,
- sequential models can consume the same normalized clip base before temporal framing.
